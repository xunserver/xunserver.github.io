# `/blog` 博客操作流程规范

本规范定义 `/blog` 命令的完整执行流程，包括命令解析、文章内容规划、指导文档读取、文件放置、GitHub 仓库写入、commit 与 push 规则。

文章的具体写作风格、内容深度、结构范式、案例使用和领域约束，必须由对应写作指导文档决定。本规范只定义操作流程和安全边界。

执行 `/blog` 时，必须按以下顺序处理：

1. 解析命令，确定模式
2. 确定文章结构和内容
3. 确定 GitHub 操作方式
4. 写入文章文件
5. 创建 commit 并 push 到远端分支
6. 总结并返回执行结果

任何关键步骤失败时，必须立即停止执行，不得继续写入、提交或推送。


# 1. 解析命令，确定模式

## 1.1 判断执行模式

`/blog` 支持两种模式：新建文章模式、修改已有文章模式。

### 1.1.1 新建文章模式

当命令中不包含 `--page` 时，必须进入新建文章模式。

命令格式：

```bash
/blog --title 标题 --tag 标签1,标签2 --category 分类 --description 描述
```

新建文章时，如果没有指定 `title`、`tag`、`category` 或 `description`，必须根据内容自动生成。

### 1.1.2 修改已有文章模式

只要命令中包含 `--page`，必须进入修改已有文章模式。

命令格式：

```bash
/blog --page some-page.md
/blog --page some-page.md --description 描述
```

如果命令中包含 `title`、`tag`、`category` 或 `description` 中任一参数，必须视为对已有文章元信息进行修改。

## 1.2 参数规则

1. 参数解析失败时，必须停止执行，并说明失败原因。
2. 修改文章时，必须确认仓库中该文件存在。
3. `--description` 用于指定 Frontmatter 的 `description` 字段。
4. 如果没有提供 `--description`，必须根据文章内容自动生成 `description`。
5. `description` 内容应概括文章主题、核心问题和主要分析方向，长度约 50 个中文字符。
6. `description` 必须是 1 段纯文本，不得包含 Markdown 图片、HTML 标签、表格、代码块或链接标记。

# 2. 确定文章结构与内容

## 2.1 VitePress Markdown 文档 Frontmatter 结构

Frontmatter 必须使用以下结构：

```md
---
title: 文章标题
category: 分类
tags:
  - 标签1
  - 标签2
date: YYYY-MM-DD
description: 一段简短的文章描述
---

{{博客具体内容}}
```

Frontmatter 规则：

1. `title`、`category`、`tags`、`description` 必须存在，并参考上一步确定的内容。
2. 新建文章时，`date` 必须使用当前日期。
3. 新建文章时，必须始终填写 `description` 字段，不依赖也不主动填写 `excerpt` 字段；如果命令提供了 `--description`，必须使用用户提供的描述，否则必须根据文章内容自动生成。
4. 修改文章时，如果命令显式提供 `--description`，必须更新 `description` 字段；如果未提供但原文章缺少 `description`，必须根据文章内容自动补齐；如果原文章已有 `description` 且用户未要求修改，必须保持不变。
5. `description` 用于 VitePress 首页文章列表展示，必须是 1 段纯文本短描述，不得包含 Markdown 图片、HTML 标签、表格、代码块或链接标记。
6. `description` 应概括文章主题、核心问题和主要分析方向，长度约 50 个中文字符，避免过长，优先控制在 80 字以内。
7. 不得添加当前博客系统不支持的 Frontmatter 字段，除非原文已有或用户明确要求。
8. 修改文章时，应尽量保持原 Frontmatter 字段顺序和未涉及字段不变。
9. 修改文章时，除非显式重新指定，未指定的字段不得主动修改；但 `description` 缺失时必须自动补齐。

## 2.2 博客内容

1. 必须根据上下文确定博客类型，并从以下模板类型中选择一个或多个规范。
2. 必须读取 GitHub 上的规范文件，并按规范文件组织博客内容结构。

- 技术类博客：`https://raw.githubusercontent.com/xunserver/xunserver.github.io/refs/heads/main/prompt/博客/文章模板/技术.md`
- 金融类博客：`https://raw.githubusercontent.com/xunserver/xunserver.github.io/refs/heads/main/prompt/博客/文章模板/金融.md`
- 杂项类博客：`https://raw.githubusercontent.com/xunserver/xunserver.github.io/refs/heads/main/prompt/博客/文章模板/金融.md`
- 每日总结博客：`https://raw.githubusercontent.com/xunserver/xunserver.github.io/refs/heads/main/prompt/博客/文章模板/每日总结.md`
- 量化交易系统架构类博客：`https://raw.githubusercontent.com/xunserver/xunserver.github.io/refs/heads/main/prompt/博客/文章模板/量化交易系统架构.md`

不确定风格时，必须使用杂项类博客，按随性记录处理。

3. 必须根据当前聊天内容组织博客内容。
4. 必须结合上方规范文件的结构，生成最终博客内容。


# 3. 仓库与分支规则

博客文章必须通过 GitHub blog 仓库创建或更新。必须使用当前上下文提供的 GitHub 操作工具。

执行规则：

1. 仅允许在 `main` 分支操作。
2. commit message 必须使用以下格式：
   - `post: 新增 xxxx 文章`
   - `post: 更新 xxx文章`
3. 不得仅在本地或临时环境中生成文章后结束。
4. 成功执行的最终结果必须写入 GitHub 仓库。
5. commit 成功后，必须 push 到远端分支。
6. 未成功 push 时，不得宣称博客已创建或已更新。
7. 不得强制推送。
8. Markdown 文章必须按照 VitePress 规则放到 `posts/{slug}.md` 中。
9. 文章图片、附件和其他静态资源只能放到 `public/posts/<slug>/` 目录中。都是放在原始 blog的仓库中，不是放在部署后仓库中
10. 仅允许操作文章相关内容。
11. 禁止改动任何其他代码。
12. 只能修改 `posts` 目录和 `public` 目录下的内容。

## 3.1 Slug 生成规则

新建文章时，必须根据标题生成 slug。

Slug 规则：

1. 尽量短、稳定、可读。
2. 由小写英文和空格组成。
3. 无法生成 slug 时，使用 `post-YYYYMMDD-HHMM`。
4. 如果 `posts/<slug>.md` 已存在，必须停止执行，并提示用户手动输入一个 slug。

# 4. 成功返回信息

成功执行后，必须返回以下信息：

- 执行模式：新建文章或修改文章。
- 文章路径。
- 资源目录路径。
- 标题。
- 分类。
- 标签。
- 描述。
- commit message。
- 推送分支。
- GitHub 文件链接或仓库路径。
- 是否已成功 push。

- xunserver/blog 是源码仓库，所有文章.md和图片资源等是放在这个仓库中。xunserver/xunserver.github.io 是vitepress 构建部署后仓库，是通过workflow/deploy 部署的，读取prompt等公开的文件是通过这个仓库暴露出的raw文件地址来实现。务必不能搞混
- 从公开仓库 xunserver/xunserver.github.io 读取流程文件
- 从公开仓库 xunserver/xunserver.github.io 读取文章模板
