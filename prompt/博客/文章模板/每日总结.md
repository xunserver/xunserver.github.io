# 每日总结生成规则

根据当天与 GPT 的对话内容，整理出一份用于博客仓库“每日记录”的 Markdown 总结。

你的任务不是写普通博客文章，而是把当天与 GPT 的有效交流内容整理成结构清晰、可追溯、可复盘的每日记录。

## 数据范围

只处理当天的对话内容。

当天指用户指定的记录日期；如果没有指定日期，则以当前日期为准。

不要主动引用其他日期的聊天内容。  
如果当天对话中明确提到了历史上下文，可以只保留与当天结论直接相关的背景说明。

## 记录目标

生成一份每日记录，用来说明今天主要和 GPT 讨论了什么、完成了什么、形成了哪些结论、还有哪些待办。

重点记录：

- 今天讨论过的主题
- 已完成的工程动作或内容动作
- 形成的判断、方案、规则或决策
- 发现的问题、原因和处理思路
- 未完成事项、后续待办和风险点

不要逐字复述聊天记录。  
不要记录无意义寒暄、重复确认、工具调用细节或临时过程噪音。

## 内容边界

如果当天对话涉及博客仓库工程维护，应只总结工程相关内容，例如：

- 仓库结构判断
- 配置、脚本、构建、部署、样式、路由等工程问题
- 修改计划、变更范围、验证结果
- PR、分支、提交策略
- 风险和回滚点

除非用户明确要求，不要改写、扩写、润色或重排博客文章正文。

如果当天对话涉及文章生成流程、提示词模板或写作规则，可以记录规则变更和目的，但不要替用户生成新的文章正文。

## 隐私和安全

不要记录以下内容：

- 密钥、Token、Cookie、密码
- 私密链接、内部访问凭据
- 完整邮箱、手机号、地址等敏感个人信息
- 不必要的账号信息
- 明显属于临时调试或无复盘价值的原始日志

如果相关信息对理解问题必要，应进行脱敏处理。

## 输出要求

使用 Markdown 输出。

语言使用中文。  
表达要简洁、准确、可复盘。  
不要使用夸张语气。  
不要添加没有依据的内容。  
无法确认的事项要明确标注“未确认”。

## 生成规则

如果当天内容较少，可以合并或省略空章节，但必须保留：

- 今日概览
- 主要主题
- 已完成事项
- 待办事项

如果当天没有可记录内容，输出：

```md
# 每日记录：YYYY-MM-DD

## 今日概览

今天没有形成需要记录的有效对话内容。

## 主要主题

无。

## 已完成事项

无。

## 待办事项

无。
```

## 质量检查

生成前检查：

- 是否只覆盖当天内容
- 是否去掉了无意义过程信息
- 是否保留了明确结论和待办
- 是否没有泄露敏感信息
- 是否没有编造未发生的事项
- 是否没有改写博客文章正文
